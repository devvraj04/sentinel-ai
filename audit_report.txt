SENTINEL AI
Complete Code Audit & Error Report
Repository: github.com/devvraj04/sentinel-ai
Audit Date: March 2026  |  Severity: 6 Critical · 13 High · 6 Medium · 1 Low

Executive Summary
What is broken and why it matters:
The ML pipeline is broken end-to-end: only 9 of 46 model features are ever written to Redis. The remaining 37 are silently zeroed out at every inference call, making the model's training AUC meaningless at runtime.
Feature evaluation is actively being skipped: the model fires only every 5th event per customer, and the force_refresh=True flag that should guarantee re-scoring is a dead no-op.
Pulse Score is semantically vague: the confidence metric is a disguised copy of PD, the sigmoid calibration constants are magic numbers disconnected from the trained threshold, and there are two conflicting threshold systems in the same codebase.
Admin credentials are hardcoded in plaintext in a public GitHub repository — this is an immediate security incident.
Three of the strongest model features (p2p_transfer_spike, credit_enquiries_3m, investment_redemption_pct) are permanently hardcoded to their 'no-stress' baseline values in the simulator.

1. Feature Pipeline — Critical Breakage
1.1  The Core Problem: 9 Features Written, 46 Features Expected
This is the most fundamental bug in the entire codebase. The feature pipeline and the ML model are operating on completely different feature sets.
What BehavioralSignals.to_feature_vector() writes to Redis:
salary_delay_days, balance_wow_drop_pct, upi_lending_spike_ratio,
utility_payment_latency, discretionary_contraction, atm_withdrawal_spike,
failed_auto_debit_count, credit_utilization_delta, drift_score
That is 9 fields.
What the LightGBM model expects (sagemaker/train.py FEATURE_COLS):
salary_delay_days, salary_amount_drop_pct, balance_wow_drop_pct,
upi_lending_spike_ratio, upi_lending_total_amount, utility_payment_latency,
discretionary_contraction, discretionary_txn_count, atm_withdrawal_spike,
atm_amount_spike, failed_auto_debit_count, failed_auto_debit_amount,
failed_utility_count, credit_utilization_delta, emi_to_income_ratio,
dpd_30_last_12m, missed_emi_streak, savings_runway_months,
revolving_utilization, credit_enquiries_3m, p2p_transfer_spike,
investment_redemption_pct, drift_salary, drift_balance, drift_lending,
drift_utility, drift_discretionary, drift_atm, drift_auto_debit,
drift_credit_card, composite_drift_score, flag_salary, flag_balance,
flag_lending, flag_utility, flag_discretionary, flag_atm, flag_failed_debit,
flag_emi_burden, flag_high_utilization, total_stress_flags, ead_estimate,
total_txn_count, total_txn_amount, income_coverage_ratio, monthly_income,
tenure_months  →  46 features total
What actually happens at scoring time (pulse_scorer.py):
fv = np.array([features.get(col, 0.0) for col in feature_cols])
37 features default to 0.0 on every single scoring call. No warning is logged. The model runs on a feature vector that is 80% zeros. This explains any observation that the Pulse Score feels unresponsive to real behavioral changes — the model literally isn't seeing them.
1.2  The Two-Pipeline Problem
There are two completely separate feature computation codepaths that produce different results:
Path A — Production (feature_pipeline.py + behavioral_signals.py):
Reads transactions from Kafka
Calls BehavioralSignalEngine.compute() → 8 behavioral signals
Writes 9 keys to Redis via to_feature_vector()
Triggers PulseScorer which reads Redis → model gets 9/46 features
Path B — Simulator (sagemaker/inference.py build_feature_vector()):
Reads transactions from PostgreSQL directly
Computes all 46 features including salary_amount_drop_pct, all drift_ fields, all flag_ fields, emi_to_income_ratio, etc.
Writes results directly to DynamoDB, bypassing the model entirely in some flows
The model was trained on the full 46-feature distribution. Production inference uses 9 features. These are different data-generating processes. The AUC measured during training does not apply to production.
1.3  _compute_drift() Calls Sub-Functions with Wrong Arguments
In behavioral_signals.py, the _compute_drift() method attempts to call each signal function with only df_hist as input:
current_val = func(df_short)
hist_val = func(df_hist) if not df_hist.empty else current_val
But functions like _upi_lending_spike(self, df_short, df_hist) and _atm_spike(self, df_short, df_hist) require two DataFrames as positional arguments. Calling them with a single arg will either:
Raise a TypeError (missing positional argument), silently caught by the except clause → signal_drifts[fname] = 0.0 for every feature
Or compute _upi_lending_spike(df_hist) treating df_hist as df_short with no baseline → wrong drift direction

2. Model Evaluation is Being Skipped
2.1  RECOMPUTE_EVERY_N_EVENTS = 5 — Real-Time Scoring is False
feature_pipeline.py line:
RECOMPUTE_EVERY_N_EVENTS = 5
def _should_recompute(self, customer_id: str) -> bool:
    return self._event_counts[customer_id] % RECOMPUTE_EVERY_N_EVENTS == 0
The feature pipeline processes every Kafka message but only triggers ML scoring on every 5th event per customer. If a customer has a single critical failed EMI event, it will not cause a score update unless 4 more transactions arrive for that same customer. The product claims real-time risk detection; the code implements batch-of-5 detection.
2.2  force_refresh is a Dead No-Op Parameter
In feature_pipeline.py:
scorer.score(customer_id, force_refresh=True)
In pulse_scorer.py, PulseScorer.score() signature:
def score(self, customer_id: str, force_refresh: bool = False) -> dict:
The force_refresh parameter is accepted but never referenced inside the method body. The intended behavior (bypass cache and always run the model) is never executed. The pipeline believes it is forcing a fresh ML score; the scorer ignores the instruction.
2.3  In-Memory Buffers Lost on Restart
The FeaturePipeline stores all transaction history in:
self._buffers: dict[str, list[dict]] = defaultdict(list)
This is pure in-memory state. When the pipeline process restarts (crash, deploy, OOM kill), all buffered history is lost. On restart, every customer's buffer is empty. The next score computed will use a near-empty transaction window, producing artificially low stress scores across the entire portfolio. There is no warm-start from Redis or PostgreSQL on initialization.
2.4  Stale Cached Scores Served Without TTL
When Redis features are unavailable, PulseScorer._last_known_score() retrieves the most recent score from PostgreSQL pulse_score_history. This score could be days or weeks old. It is returned with cached: True but there is no maximum staleness check. A customer whose financial situation changed dramatically last week can be served a 30-day-old 'green' score.

3. Pulse Score is Vague and Unreliable
3.1  Two Conflicting Threshold Systems
The codebase has two completely separate, inconsistent threshold definitions:
Any component that reads settings.pulse_score_high_risk for tier decisions will classify a customer with score 70–74 as 'orange' while the scoring engine classifies them as 'red'. The discrepancy is 5 points wide at every tier boundary.
3.2  Sigmoid Calibration Constants Are Disconnected from Model
scoring_utils.py:
_SIGMOID_CENTER = 0.42   # claimed to be the model's decision threshold
_SIGMOID_K      = 8.0    # steepness
train.py computes the ECL-optimal threshold dynamically:
threshold, min_ecl = select_ecl_threshold(y.values, prob, ead)
But this computed threshold is saved into lgbm_model.joblib as package['threshold'] and never read back by scoring_utils.py. The sigmoid center is hardcoded to 0.42 regardless of what threshold the trained model actually produced. If the optimal threshold from a retrain shifts to 0.38 or 0.50, the entire Pulse Score scale shifts silently without anyone noticing.
3.3  Confidence Score is a Disguised Copy of PD
pulse_scorer.py:
confidence = float(abs(pd_prob - 0.5) * 2)
This is mathematically identical to: confidence = |2 × pd_prob − 1|. It equals 0.0 when PD=0.50 and 1.0 when PD=0.0 or PD=1.0. It tells you nothing about model uncertainty, calibration quality, or feature reliability — it is just the signed distance from the classification boundary. A model that is wildly overfit and outputs PD=0.99 gets confidence=0.98, even if that PD is completely wrong.
A genuine confidence metric would use: posterior variance from calibration curves, SHAP value variance, or Monte Carlo dropout if using a neural model. For LightGBM, isotonic regression or Platt scaling can produce calibrated probabilities whose distance from 0/1 reflects true confidence.
3.4  realtime_stress_tracker.py Score Impacts are Invented
The stress tracker assigns fixed score_impact strings to transaction types:
'failed_auto_debit':  score_impact: '+8'
'lending_app_upi':    score_impact: '+7'
'large_atm_withdrawal': score_impact: '+4'
These numbers are not from the model. They are not from SHAP. They are hardcoded constants in a display dict. Meanwhile the customers.py score-impact API correctly uses actual deltas from pulse_score_history. The tracker contradicts the API — two parts of the same product show different score impacts for the same events.

4. Hardcoded Values — Complete List
Security: Hardcoded Admin Credentials
api/routers/auth.py contains:
if form.username == 'admin@sentinel.bank' and form.password == 'sentinel_admin':
This is in a public GitHub repository. The bcrypt CryptContext is imported and initialized but never used for the admin check. The fix requires removing the hardcoded block and implementing a proper database lookup with bcrypt.verify:
user = await db.fetch_one('SELECT * FROM users WHERE email = $1', form.username)
if not user or not pwd_context.verify(form.password, user.hashed_password):
    raise HTTPException(401, 'Invalid credentials')
