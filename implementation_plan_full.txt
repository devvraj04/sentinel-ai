1. Executive Summary
1.1 The Problem
Banks globally are caught in a reactive delinquency cycle. Traditional collections systems trigger only after a customer has missed a payment — typically 30+ Days Past Due (DPD). At that stage, the Expected Credit Loss (ECL) has already materialized, the customer relationship is strained, and collection costs — ranging from 15–20% of the recovered amount — have begun to accumulate.

The core insight driving Sentinel is straightforward but powerful: financial distress does not begin at the first missed payment. It begins weeks earlier, embedded in subtle behavioral signals — a salary arriving 5 days late, a creeping balance on a lending UPI app, utilities paid in the last 2 days of a billing cycle, and a gradual drawdown of savings. These signals are individually weak but collectively definitive.


1.2 Sentinel's Answer
Sentinel is a real-time, open-source Pre-Delinquency Intervention Engine that:
Ingests transaction events in real-time via Apache Kafka (< 50ms end-to-end latency).
Detects behavioral drift by comparing 7–14 day rolling behavioral windows against each customer's personal historical baseline.
Scores every customer continuously using a dual-model stack: LightGBM (structured signals) + LSTM (temporal sequences), producing a Pulse Score (0–100).
Triggers tiered, empathetic interventions (payment holiday, flexible EMI, digital nudge) before the first payment is missed.
Provides full SHAP-based explainability for every risk score, with immutable audit logs meeting RBI, IFRS 9, and Basel III requirements.

2. System Architecture Overview
2.1 Architectural Philosophy
Sentinel is built on three core architectural principles that distinguish it from legacy delinquency detection systems:

Event-Driven, Not Batch-Driven — every customer transaction immediately flows into the scoring pipeline. There is no overnight batch job creating a 24-hour detection blind spot.
Personalized Baseline, Not Global Threshold — risk is measured against each customer's own historical behavioral pattern, not a generic population threshold. This dramatically reduces false positives.
Explainability by Design, Not by Afterthought — SHAP feature attribution is computed alongside every Pulse Score, not retrofitted. Every decision is audit-ready from day one.

4.3 Apache Flink — Stream Processing Jobs
Apache Flink provides stateful, exactly-once stream processing with event-time semantics and watermarking — critical for accurate behavioral window computations even in the presence of late-arriving events.

4.3.1 Job 1: Transaction Enrichment Job
Input: sentinel.raw.transactions Kafka topic
Processing: Enriches each transaction with customer profile data (income, loan obligations, credit limit) via an async lookup to Redis (customer profile cache, TTL 24h).
Windowing: No windowing — pure event-time enrichment with a 30-second event-time watermark for late arrivals.
Output: sentinel.enriched.events topic (Avro encoded)
Parallelism: 16 task slots (scales to 64 at peak)
State backend: RocksDB (persistent, incremental checkpoints to S3/MinIO every 60 seconds)

4.3.2 Job 2: Behavioral Feature Computation Job
This is the most critical Flink job. It maintains per-customer stateful windows and computes all behavioral drift signals.

Input: sentinel.enriched.events
Key by: customer_id (ensures all events for a customer go to the same task slot)
Windows computed:
7-day tumbling window: salary credits, ATM withdrawals, UPI lending transfers, discretionary spend totals
14-day sliding window (slides every 1 day): credit utilization rolling average, balance trend
30-day session window (gap 24h): utility payment timing pattern detection
State management: Each customer maintains a circular buffer of 90 days of daily aggregates in Flink's RocksDB state backend.
Output: Writes computed features to Feast online store (Redis) AND sentinel.features.customer Kafka topic.
Late data handling: Events arriving up to 1 hour late are accepted and state is recomputed.

4.3.3 Job 3: Salary Timing Drift Detector
A specialized Flink Complex Event Processing (CEP) job.
Pattern: Detects 'SALARY_CREDIT not received within expected_salary_day + 3 days' for any customer with a known employer.
Uses Flink CEP library with a PATTERN that matches: [salary_expected_window passes] -> [no salary event received] -> [emit drift alert].
Output: Enriches feature vector with salary_delay_days = current_day - expected_salary_day.

6. ML Model Design & Training
6.1 Model Architecture — Dual-Model Ensemble
Sentinel uses a two-model ensemble approach. Each model captures a different aspect of customer risk, and their outputs are combined through a meta-learner that is calibrated to minimize Expected Credit Loss (ECL), not raw accuracy.


6.1.1 Model A: LightGBM — Structured Behavioral Risk Classifier

6.1.2 Model B: LSTM — Temporal Sequence Risk Model
The LSTM model processes each customer's transaction history as a time-ordered sequence, learning behavioral patterns that unfold over weeks — patterns that a snapshot-based model cannot detect.

6.2 Training Data Pipeline
6.2.1 Label Generation
Positive label (delinquency = 1): Customer had DPD ≥ 1 in a 30-day window following the feature snapshot date.
Negative label (no delinquency = 0): Customer had zero DPD in the 30-day window following the feature snapshot.
Label horizon: Features are computed at T=0, label is observed at T+14 to T+28 days. This simulates the 2–4 week early detection target.
Label leakage prevention: Strict temporal split — training data uses features computed before a cutoff date; validation and test data use features computed after. No future information is used in feature computation.

6.2.2 Class Imbalance Handling
Typical delinquency rate: 2–5% of portfolio. This creates extreme class imbalance.
Techniques applied (in order of application):
SMOTE (Synthetic Minority Oversampling Technique): Generate synthetic minority class samples in feature space using k-nearest-neighbors interpolation. Applied only on training set, never on validation or test sets.
Tomek Links: Remove majority class samples that are closest to minority class (borderline cleanup after SMOTE).
Class weights: scale_pos_weight = (n_negative / n_positive) in LightGBM; pos_weight tensor in PyTorch LSTM.
Focal Loss (optional): For extreme imbalance cases, replace BCE with Focal Loss (gamma=2, alpha=0.25) to focus learning on hard-to-classify minority examples.

6.2.3 Training Infrastructure
Offline training: Apache Airflow DAG triggers monthly full retraining using 18 months of historical data from the Feast offline store (PostgreSQL + Parquet).
Point-in-time correct feature retrieval: Feast's get_historical_features() ensures features are retrieved as they appeared at the label timestamp — preventing data leakage.
Training environment: Docker container with GPU support (pytorch:2.x-cuda11.8), launched by Airflow on a GPU-enabled machine (or Kubernetes job with GPU node selector).
Hyperparameter tuning: Optuna with TPE sampler, 100 trials, 5-fold time-series cross-validation (TimeSeriesSplit from scikit-learn). Early stopping patience = 50 rounds.
Experiment tracking: All hyperparameter combinations, metrics, and artifacts (model weights, SHAP plots, confusion matrices) logged to MLflow Tracking Server.

7. Model Serving & Real-Time Inference
7.1 Inference Architecture
The model serving layer must meet a strict <50ms latency SLA for the end-to-end scoring pipeline (feature retrieval + model inference + SHAP computation + score storage). This is achieved through:
BentoML service for LightGBM: Compiled to ONNX format for 3–5x speedup over native Python inference. Served via FastAPI-based BentoML runner.
TorchServe for LSTM: PyTorch model exported to TorchScript (.pt format). Served via TorchServe with custom handler that handles sequence padding and attention mask generation.
Feast online feature retrieval: Redis-backed feature lookup with connection pooling. <5ms P99.
SHAP computation: LightGBM Tree SHAP (native C++ implementation). Computed in-process alongside inference. <5ms for 47 features.
Asynchronous score persistence: Score written to Redis synchronously (for immediate dashboard refresh). PostgreSQL write is async (Celery task) to avoid blocking the scoring path.

8. Explainability & Fairness Framework
8.1 SHAP — SHapley Additive Explanations
SHAP is the mathematical backbone of Sentinel's explainability layer. It is grounded in cooperative game theory: each feature is treated as a player in a coalition game, and its Shapley value represents its fair contribution to the model's prediction, averaged across all possible feature orderings.


8.2 SHAP Implementation Strategy
8.2.1 LightGBM — TreeExplainer (Fast Path)
Uses TreeExplainer from the shap library — optimized C++ implementation with O(TLD) complexity (T=trees, L=leaves, D=depth).
Computes exact SHAP values (not approximations) for every tree in the LightGBM ensemble.
Output: 47-dimensional SHAP value vector per customer, where each value represents the feature's signed contribution to the log-odds of default.
Performance: < 5ms for 47 features with 1000-tree LightGBM model.

8.2.2 LSTM — DeepExplainer (Temporal Attribution)
Uses shap.DeepExplainer or shap.GradientExplainer for PyTorch LSTM models.
Generates temporal SHAP values: attribution matrix of shape (90_days × 12_daily_features), showing which days and which features contributed most to the risk prediction.
Performance: 50–200ms — computed asynchronously (not on the critical scoring path). Results cached for 24h per customer.

8.3 Fairness Monitoring — Adverse Impact Ratio (AIR)
Sentinel's fairness framework is built around the Adverse Impact Ratio (AIR), which is the standard metric used by the US Equal Credit Opportunity Act (ECOA) and Fair Credit Reporting Act (FCRA), and is directionally applicable to RBI's Fair Practice Code for NBFC/Banks.



8.4 Population Stability Index (PSI) — Model Drift Detection
The Population Stability Index (PSI) measures how much the distribution of model input features (or output scores) has shifted between the training period and the current production period. A rising PSI indicates that the model is being asked to score a population it has not seen before — the signal that retraining is required.


PSI is computed daily by an Airflow DAG (sentinel_psi_monitor_dag) using Evidently AI's PSI report.
Monitored for: all 47 input features AND the Pulse Score distribution (output PSI).
Output PSI > 0.25 triggers an automatic MLflow model retraining pipeline.
Feature PSI reports are stored in MLflow artifacts and displayed on the Sentinel Ops Dashboard (Grafana).

9. Intervention Engine & Messaging
9.1 Risk Tiering & Intervention Mapping
Every customer with an active loan or credit obligation is continuously scored. When a Pulse Score breaches a tier threshold, the Intervention Engine activates the appropriate response workflow.


9.2 Message Generation — Template Engine + GenAI
Sentinel combines structured Jinja2 message templates (for consistency and compliance) with a GenAI layer (optional — open-source LLM via Ollama or OpenAI API) for personalization.

9.2.1 Jinja2 Base Template (Tier 1 — Critical)

9.2.2 GenAI Personalization Layer (Optional — Ollama / Llama 3)
When Pulse Score confidence interval is narrow (model uncertainty is low), a local LLM (Llama 3 8B via Ollama) generates a personalized variant of the base template.
System prompt constraints are strictly enforced: no risk language, single sentence, specific tone, CTA required, character limit.
All GenAI-generated messages are validated against a regex rule engine before dispatch. Any violation falls back to the Jinja2 base template.
All generated messages are logged with their source (template vs. GenAI) and A/B variant ID for performance analysis.

9.3 Closed-Loop Automation — Customer Response Handling
When a customer responds 'YES' to an intervention message, the following automated workflow executes within 5 seconds:

Webhook receiver (FastAPI endpoint) captures the 'YES' reply from WhatsApp Business API or SMS gateway.
Customer identifier resolved from phone number → account mapping in Redis.
Eligibility check: LMS API queried for current DPD status, existing active deferrals, and deferral count in last 12 months.
If eligible: LMS API called to patch EMI schedule (defer by 30 days). Status returned.
If not eligible (e.g., already in DPD, deferral limit exceeded): Alternative offer generated (payment plan, financial counseling referral).
Confirmation message sent to customer with revised EMI schedule.
Intervention outcome logged: customer_id, intervention_id, response, eligibility_result, outcome, timestamp → PostgreSQL intervention_outcomes table.
Feedback event published to Kafka (sentinel.interventions.responses) → triggers model feedback loop in Airflow.
Pulse Score re-evaluated 24h after intervention; monitoring continues.

11.2.2 Model Registry & Promotion Workflow

11.2.3 Champion/Challenger Framework
When a new Challenger model passes the Staging gate, it is deployed in shadow mode (scores all customers but does not trigger interventions).
After 7 days of shadow scoring, a statistical comparison is performed: two-sample KS test on Challenger PD vs. Champion PD distributions; recall@top15% on labeled outcomes (using 7-day delayed label assignment).
If Challenger shows ≥ 3% KS improvement with p-value < 0.05: promote to production. The transition happens via BentoML's canary deployment — 10% of traffic routes to Challenger for 24h, then 100%.

17.2 Scalable Infrastructure (18–24 months)
Edge-based risk scoring: Deploy lightweight quantized LightGBM model (ONNX quantized to INT8) directly to ATM firmware and mobile banking SDK — local Pulse Score computation with sub-millisecond latency, no network round-trip.

 | Why Ensemble?
LightGBM excels at detecting snapshot-based structured risk signals (e.g., today's credit utilization vs. baseline). LSTM excels at detecting temporal sequence patterns (e.g., the progressive 6-week degradation pattern before a default). Neither alone captures the full risk picture. The ensemble captures both.

 | BentoML Service Structure (sentinel_lgbm_service.py)
import bentoml
from bentoml.io import NumpyNdarray, JSON
import shap, numpy as np

lgbm_runner = bentoml.lightgbm.get("sentinel_lgbm:latest").to_runner()
svc = bentoml.Service("sentinel_lgbm_scorer", runners=[lgbm_runner])

@svc.api(input=JSON(), output=JSON())
async def score(features: dict) -> dict:
    feature_array = np.array([list(features.values())], dtype=np.float32)
    pd_score = await lgbm_runner.predict_proba.async_run(feature_array)
    
    # Compute SHAP values using pre-loaded Tree explainer
    explainer = shap.TreeExplainer(lgbm_runner._model)
    shap_values = explainer.shap_values(feature_array)[1]
    
    top_drivers = sorted(
        zip(features.keys(), shap_values[0]),
        key=lambda x: abs(x[1]), reverse=True
    )[:5]
    
    return {
        "pd": float(pd_score[0][1]),
        "pulse_score": round(float(pd_score[0][1]) * 100),
        "top_shap_drivers": [{"feature": f, "shap": float(v)} for f, v in top_drivers],
        "model_version": "lgbm_v2.1.0"
    }

 | The SHAP Promise for Banking Compliance
SHAP satisfies the three regulatory properties required for credit decision explanations: (1) Efficiency — all feature contributions sum to the prediction gap from the baseline. (2) Symmetry — features contributing equally receive equal values. (3) Dummy — features with no effect receive zero contribution. These properties make SHAP a de facto standard for GDPR 'right to explanation' and ECOA/FCRA adverse action notice compliance.

 | AIR Formula
AIR = (Favorable outcome rate for protected group) / (Favorable outcome rate for reference group)

Regulatory threshold: AIR ≥ 0.80 (the 4/5ths rule). If AIR < 0.80, it indicates the model has an adverse disparate impact on the protected group.

In Sentinel's context: 'favorable outcome' = NOT being flagged as high-risk. The metric is computed across geographic (rural/urban), age (≤30, 31–50, 51+), and gender segments.

Fairness Dimension | Protected Group | Reference Group | Monitor Frequency | Action If AIR < 0.80
Geographic | Rural customers | Urban customers | Weekly | Recalibrate geographic_risk_tier feature weights; audit feature set for geographic proxies
Age | Customers ≤ 30 years | Customers 31–50 years | Weekly | Remove age-correlated features; apply sample reweighting in training
Gender | Female customers | Male customers | Weekly | Remove gender-correlated features; apply adversarial debiasing training
Income Band | Bottom 25% income | Median income band | Weekly | Adjust emi_to_income_ratio feature normalization; increase SMOTE weight for low-income class

 | PSI Formula
PSI = Σ [(Actual_i - Expected_i) × ln(Actual_i / Expected_i)]

where i = each bin of the feature or score distribution.

Interpretation: PSI < 0.10 → Stable (no action). PSI 0.10–0.25 → Moderate shift (investigate). PSI > 0.25 → Significant shift (trigger retraining).

 | Drift Score Formula
For feature f, at time T:

μ_short(f,T) = (1/k) × Σ[t=T-k to T] f_t          (Rolling short-term mean, k=7 days)

μ_hist(f,T)  = (1/m) × Σ[t=T-m to T-k] f_t         (Historical baseline mean, m=90 days)

σ_hist(f,T)  = std deviation of f over historical window (T-m to T-k)

Drift_f(T)   = (μ_short(f,T) - μ_hist(f,T)) / σ_hist(f,T)

Early Stress Flag: |Drift_f(T)| > θ_f
where θ_f is the per-feature drift threshold (see Feature Catalog, Section 5.2)

Composite Drift Score = Σ_f [ w_f × |Drift_f(T)| × sign_f ]
where w_f = SHAP-derived feature importance (updated weekly)
and sign_f = +1 if drift direction is 'stress-positive', -1 otherwise

 | IFRS 9 ECL Formula
ECL = PD × LGD × EAD × DF

Where:
  PD  = Probability of Default (output of Sentinel's LightGBM meta-learner, calibrated)
  LGD = Loss Given Default (bank-specific, e.g., 40% for unsecured personal loans, 20% for secured home loans)
  EAD = Exposure At Default (outstanding principal + undrawn committed credit lines)
  DF  = Discount Factor (risk-free rate discounting for lifetime ECL; typically 1 for 12-month ECL Stage 1)

Risk Prioritization for Intervention:
  Priority_i = PD_i × EAD_i    (Note: LGD omitted — all same product type in same tier)
  High exposure + high risk → first in intervention queue

Model threshold is selected to minimize portfolio-level ECL, not to maximize AUC or accuracy.
Threshold selection: argmin_θ [ Σ_i ECL_i | scored_high_risk(i) = 1 ]

 | Pulse Score Derivation
Step 1: LightGBM produces PD_lgbm ∈ [0, 1]
Step 2: LSTM produces PD_lstm ∈ [0, 1]
Step 3: Meta-learner (Logistic Regression + Platt Scaling):
        PD_final = σ(w1 × logit(PD_lgbm) + w2 × logit(PD_lstm) + w3 × x_salary_delay + w4 × x_emi_to_income + ...)
        where logit(p) = ln(p / (1-p)) and σ(z) = 1 / (1 + e^(-z))
Step 4: Pulse Score = round(PD_final × 100)  → integer in [0, 100]

Risk Tier Assignment:
  Pulse Score ≥ 75  → Tier 1 (CRITICAL)   → Immediate intervention
  Pulse Score 55–74 → Tier 2 (HIGH RISK)  → EMI flexibility offer
  Pulse Score 40–54 → Tier 3 (MODERATE)  → Digital nudge
  Pulse Score 25–39 → Tier 4 (WATCH)     → In-app alert
  Pulse Score 0–24  → Tier 5 (STABLE)    → Passive monitoring

 | PSI Formula
PSI = Σ_i=1_to_N [ (Actual_i - Expected_i) × ln(Actual_i / Expected_i) ]

Where:
  N = number of bins (typically 10 equal-frequency bins from training distribution)
  Actual_i = fraction of production data in bin i
  Expected_i = fraction of training data in bin i

Interpretation:
  PSI < 0.10   → No significant change (model stable)
  0.10 ≤ PSI < 0.25 → Moderate change (investigate; consider retraining)
  PSI ≥ 0.25   → Significant change → Automatic retraining trigger

Implementation: Evidently AI PSIReport + custom Airflow operator

 | AIR Formula
AIR = Rate_protected / Rate_reference

Where Rate = P(Favorable outcome | group)
In Sentinel: Favorable outcome = Pulse Score < 75 (NOT flagged as high-risk)

AIR = P(Pulse < 75 | protected_group) / P(Pulse < 75 | reference_group)

Regulatory threshold: AIR ≥ 0.80 (the 4/5ths rule from EEOC Uniform Guidelines)

Example:
  If 70% of urban customers have Pulse < 75 (not flagged),
  and 55% of rural customers have Pulse < 75 (not flagged),
  AIR = 55% / 70% = 0.786 → BELOW THRESHOLD → bias review required

Term | Definition
AIR (Adverse Impact Ratio) | Ratio of favorable outcome rates between a protected group and a reference group. Regulatory threshold ≥ 0.80 (4/5ths rule). Used to detect model bias.
AUC / AUROC | Area Under the Receiver Operating Characteristic Curve. Measures discrimination ability of a binary classifier. Range [0.5, 1.0]; higher is better.
AUPRC | Area Under the Precision-Recall Curve. More informative than AUROC for imbalanced datasets (like delinquency detection).
Behavioral Drift | A statistically significant deviation of a customer's recent behavioral pattern from their own historical baseline. The core signal that Sentinel detects.
Brier Score | Mean squared error of probabilistic predictions. Measures calibration. Range [0, 1]; lower is better. Brier = 0 means perfect calibration.
CDC (Change Data Capture) | Technology (Debezium) that captures row-level database changes and streams them to Kafka in real-time.
ECL (Expected Credit Loss) | IFRS 9 formula: PD × LGD × EAD. The probability-weighted estimate of credit losses. Banks must provision this amount.
EAD (Exposure at Default) | The amount outstanding that could be lost if a customer defaults. Typically the outstanding principal plus undrawn committed lines.
Feast | Open-source Feature Store. Manages feature definitions, computes features, serves them at low latency, and ensures training-serving consistency.
Flink CEP | Apache Flink Complex Event Processing. A library for detecting patterns of events in data streams (e.g., salary not received within expected window).
HITL (Human-in-the-Loop) | A system design where automated model outputs are reviewed by a human before consequential actions. Required for high-stakes AI decisions.
KS Statistic | Kolmogorov-Smirnov statistic. Measures the maximum separation between the true positive rate and false positive rate curves. Primary credit model metric.
LGD (Loss Given Default) | The fraction of EAD that would be lost if a customer defaults. Varies by product: typically 40% for unsecured, 20% for secured.
LightGBM | Light Gradient Boosting Machine. An efficient gradient boosted decision tree framework optimized for tabular data. Best-in-class for structured financial risk scoring.
LSTM | Long Short-Term Memory. A type of Recurrent Neural Network (RNN) that can learn from sequential time-series data. Used in Sentinel for temporal pattern detection.
MLflow | Open-source platform for ML experiment tracking, model versioning, registry, and deployment.
ONNX | Open Neural Network Exchange. A framework-agnostic ML model format. Converting LightGBM to ONNX enables 3–5x faster inference via ONNX Runtime.
PD (Probability of Default) | The model's estimate of the probability that a customer will miss a payment within the 14–28 day prediction horizon. Core Sentinel output.
PSI (Population Stability Index) | Measures distributional shift between training data and production data. PSI > 0.25 triggers automatic model retraining.
Pulse Score | Sentinel's branded customer risk score: PD × 100, rounded to integer [0–100]. 0 = stable, 100 = maximum risk.
SHAP | SHapley Additive Explanations. A game-theory-based method that computes each feature's contribution to a model's prediction. Gold standard for ML explainability.
SMOTE | Synthetic Minority Oversampling Technique. Generates synthetic examples of the minority class (delinquent customers) to address extreme class imbalance in training data.
Training-Serving Skew | A critical ML failure mode where features are computed differently in training vs. production, causing models to underperform. Feast prevents this by enforcing a single feature computation pipeline.
Waterfall Chart (SHAP) | A visualization showing each feature's contribution to the Pulse Score, displayed as a waterfall from the base value to the final prediction. Used in the Customer Profile dashboard.
